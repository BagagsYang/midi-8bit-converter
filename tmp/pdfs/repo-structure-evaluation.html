MIDI-8bit Repository Evaluation

Scope: classify the current repository and decide whether it should stay together or be split, using only repo evidence.

Final verdict: Keep as monorepo, but reorganise.

1. Repository classification

Current state: incomplete / poorly structured hybrid.

It is not a clean monorepo because the root has no shared build entrypoint, no root workspace, and only one platform CI workflow. It is not just a loose folder dump either, because html-app and macos-app directly depend on shared/midi_to_wave.py, and windows-app uses the same Python code as its parity oracle.

2. Evidence from structure

  • html-app/: Python Flask web app, not Node. Evidence: html-app/app.py, templates/, static/, no package.json or pyproject.toml.
  • macos-app/: Xcode project, not a Swift Package. Evidence: MIDI8BitSynthesiser.xcodeproj/project.pbxproj, SwiftUI sources, custom build phase named Bundle Python Helper.
  • windows-app/: .NET 8 WinUI 3 solution with xUnit tests and Inno Setup installer. Evidence: Midi8BitSynthesiser.sln, *.csproj, installer/Midi8BitSynthesiser.iss, Directory.Packages.props.
  • shared/: plain Python module plus dependency file, not an installable package. Evidence: shared/midi_to_wave.py, shared/requirements.txt, no packaging metadata.
  • desktop-app/: no files or manifest in this checkout; responsibility is unclear and it is absent from the root variant matrix.
  • Root: global.json only pins .NET; there is no root Package.swift, workspace, Makefile, or monorepo task runner.

3. Problems found

  • Shared code is only partly shared: html-app/app.py imports shared directly, macos-app/macos/build_desktop_resources.sh freezes the same script with PyInstaller, but windows-app/src/Midi8BitSynthesiser.Core/SynthesisRenderEngine.cs is a separate C# implementation.
  • Dependency boundaries are blurred: shared/requirements.txt includes core DSP dependencies plus Flask and PyInstaller, so the "shared" layer also owns web and build-packaging dependencies.
  • Assets are duplicated/inconsistently sourced: Windows stores preview WAVs in windows-app/src/.../Assets/Previews, while macOS copies them from html-app/static/previews. The files are byte-identical, so there is duplicate source of truth.
  • CI/CD is platform-skewed: only .github/workflows/windows-release.yml exists. There is no comparable macOS or HTML verification pipeline.
  • Repo hygiene is weak: tracked Xcode derived data exists under macos-app/.xcodebuild/.

4. Options considered

Keep as-is: lowest short-term cost, but the current ambiguity will keep increasing drift. The Windows rewrite, Python reference, and duplicated assets will remain awkwardly coupled.

Split now: clean ownership boundaries, smaller repos, independent release cadence. Cost: you would have to version and publish the Python reference core first, replace direct path dependencies, duplicate or publish shared assets, and redesign the Windows parity tests that currently locate ../../../shared.

Keep one repo, but reorganise: preserves atomic cross-platform changes to the render contract while fixing the structural debt. This is the best fit for the repo as it exists today.

5. Recommended structure

Target layout:

apps/web-flask/ apps/macos/ apps/windows/

core/python-renderer/ assets/previews/ docs/ .github/workflows/

  • Move platform apps under apps/ and move the Python renderer to core/python-renderer/.
  • Turn the Python renderer into a real package with scoped dependencies. Keep core deps there; put Flask in the web app and PyInstaller in the macOS build layer.
  • Move preview WAVs to assets/previews/ so web, macOS, and Windows all consume one source of truth.
  • Keep the Windows C# engine in the repo, but treat it explicitly as a second renderer implementation validated against the Python reference.
  • Add per-app CI: at minimum HTML smoke/import test, macOS build validation, and Windows build/test/release.

6. Trade-offs

  • Why not split now: the real coupling is still high. Path-based imports, build-time bundling, shared parity tests, and shared assets mean a split would add release and package-management overhead before it reduces complexity.
  • Why reorganise instead: one repo still makes cross-platform contract changes tractable, but the repo should reflect architecture instead of historical accretion.
  • When a split would become reasonable: once the Python renderer is a versioned package, the asset pipeline is centralised, and each platform has independent CI/release ownership.

7. Summary

This repository is one product with three delivery paths, not three unrelated products. The problem is not that it is a monorepo; the problem is that it is an under-structured monorepo. Keep it together, remove the accidental coupling, and make the shared/core versus platform/app boundary explicit.
